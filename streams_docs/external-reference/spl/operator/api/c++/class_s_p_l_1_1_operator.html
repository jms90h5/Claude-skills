Detailed Description

Class that serves as the base class of all SPL primitive operators.

Primitive C++ operators extend this class to implement their processing logic.

The life-time of the operator starts with a call to its constructor. It is common for an operator to perform initialization tasks in the constructor.

Next, the allPortsReady function is called by the runtime to notify the operator that all PE ports are ready and it is safe to submit tuples. This function is often used to by the operator to create threads (either using the createThreads function or by other means) that will perform asynchronous work or in the case of source operators, do the main work of the operator.

Once all PE ports are ready, the tuple and punctuation process functions (process(Tuple) and process(Punctuation)) of the operator are called by the runtime to drive the execution. These functions may run concurrently and thus proper synchronization should be performed (see AutoPortMutex) by the operator developer. It is safe to submit tuples from within the context of the tuple and punctuation processing functions, and there is no reason to wait for the allPortsReady notification to do so.

The prepareToShutdown function is called by the runtime to notify the operator about the pending termination of the PE. Commonly, this function is used by the operator to make a note of the termination, so that any asynchronous processing performed by the operator is notified properly of the pending shutdown. The tuple and punctuation processing functions can execute concurrently with the prepareToShutdown function. See also: ProcessingElement::getShutdownRequested.

Finally, the destructor is called to finalize the operator. The destructor is commonly used by the operator to perform resource cleanup tasks. It is guranteed that no tuple or punctuation processing functions are active when the destructor is running.

The common flow of the operator life-cycle is given as follows:

Control path Data path (async.)
============================================
Constructor
| \
allPortsReady process(Tuple,..)
| and
prepareToShutdown process(Punctuation,...)
| /
Destructor

If an operator is configured to checkpoint in its SPL invocation, then the restoreCheckpoint function is called after the constructor in case the operator is restarted and there is a checkpoint to restore from. Also, when checkpointing is configured, getCheckpoint function is called by the runtime to asynchronously checkpoint the operator.

virtual void SPL::Operator::checkpointStateVariables ( NetworkByteBuffer &  opstate ) const
pure virtual

This function is automatically generated by the SPL compiler, in order to allow checkpointing of SPL logic state variables. When called, this routine will save the current values of the logic state variables into opstate. This function should be called from getCheckpoint to save any state variables.

Parameters
